重構是 TDD 開發流程的重頭戲，當第一次 綠燈 之後，就是不斷的重構，一直維持 綠燈，讓程式碼架構更好，卻又不違背原來的需求，而PhpStorm 提供強悍的重構功能，讓我們在重構時不需要 copy and paste，可直接快速重構，提升開發效率。

Motivation

在如何使用 PhpStorm 實現 TDD、重構與偵錯?，曾經以重構–改善既有程式的設計第一章的範例，實際以 TDD 與重構的方式，完全在 PhpStorm 執行，不過當時並沒有使用到 PhpStorm 所有的重構功能，本文則實際深入探討 PhpStorm 全部的重構功能。

Extract Method

在 TDD 第一個 綠燈 之後，為了，實務上我們會使用 Extract Method 重構，讓程式碼可讀性更高，語意更清楚。

PostService.php12 12GitHub Commit : 顯示所有有 Description 的 Title

為了顯示全部有 description 的 title，需要在 foreach 內對 description 做判斷，並新增至 $titles 陣列。

其中 if () 這段程式碼，包含了底層的 if() 判斷，還包含了新增至陣列的實作，看起來比較突兀，也就是我們應該 把抽象出來的方法放在一起，而不要把底層程式碼和抽象出來的方法放在一起 的概念。

所謂 抽象出來的方法是指 Extract Method 後的產物，它代表著這段程式碼的意圖。

每一階程式碼都有它放在這裡的意義，千萬不要將高階和低階的方法混雜在一起，這樣讀程式碼時會很突兀。

選擇要 Extract Method 的程式碼，按熱鍵 ⌃ + T，選擇 Method...。

  • Name : 輸入 Extract Method 後新的 method 名稱，輸入 filterTitlesWithDescription。
  • Visibility : 選擇 Private。
  • Return output variable(s) through : 選擇 Return statement。

PostService.php13 13GitHub Commit : 重構成 filterTitlesWithDescription

PhpStorm 會自動幫我們建立了 filterTitlesWithDescription()，並且自動幫我們修改了呼叫 filterTitlesWithDescription() 的方式。

對於 showTitlesWithDescription() 來說，其職責在於 foreach() 去顯示資料，而 filterTitlesWithDescription() 的職責在於過濾資料，如此程式碼的可讀性更高，語意也更明顯，也符合 SOLID 的單一職責原則。

Extract Method 是重構當中，最常用的手法，而 PhpStorm 的 Extract Method 功能非常強大。

Change Signature

包含了以下多項重構功能 :

  • 改變 method 名稱
  • 新增參數
  • 刪除參數
  • 更改參數的預設值
  • 調整參數順序
  • 修改參數名稱
  • 新增一系列 function call 的參數

當使用 Change Signature 時，PhpStorm 會搜尋所有使用此 method 之處，並做相對應的修改。

PostRepository.php

PostRepository 的 getTitle() 目前參數為 $id 與 $default，想新增一個參數為 $content。

目前 PostController 呼叫 PostSerive，而 PostService 呼叫 PostRepository，若 PostRepository 的 getTitle() 增加參數，勢必 PostService 與 PostController 也必須修改。

將滑鼠游標放在 getTitle()，按熱鍵 ⌃ + T，選擇 Change Signature...。

  • Name : 重構成新的 method 名稱，若不改變可不修改。
  • 新增參數 : 按熱鍵 ⌘ + N 新增參數 $content，後面的 Default 表示其他相依的 method，該新增什麼變數名稱。

  • Propagate 參數 : 按熱鍵 ⌥ + G。
  • 選擇 Method : 選擇要 propagate 參數的 method。

Extract Interface

TDD 要求我們在設計階段，先根據當前的需求 綠燈 就好，避免因為 over design 而墊出一堆 interface，而讓架構提早複雜化，但真的需求來時，有必要墊 interface 還是要墊，此時我們可以使用 Extract Interface 將目前原有的 class 抽出 interface。

原本 PostRepository 並沒有墊 interface，但因為有切換 MySQL 與 MongoDB 的需求，只好將 repository 加以抽象化，因此想定義 PostRepositoryInterface。

將滑鼠游標放在 PostRepository，按熱鍵 ⌃ + T，選擇 Interface...。

  • Interface Name : 輸入你想要使用的 interface 名稱，輸入 PostRepositoryInterface。
  • 所有依賴注入都改成 Interface : 將 Replace class references with interface where possible 打勾。
  • Namespace : 若你想一併改變 interface 的 namespace，可在此修改，如有人可能會把所有 interface 專門放在一個 namespace 下。
  • Target destination directories : 為了符合 PSR-4，namespace 與 directories 會一致。
  • Members to form interface : 選擇你要抽出成為 interface 的 method。
實務上不建議將 class 的所有 method 原封不動抽成 interface，應該以需求的觀點去考量 interface，並遵守 SOLID 的 ISP 的介面隔離原則，如此才能符合 SRP 單一職責原則與 DIP 依賴反轉原則的要求，最後達成 OCP 開放封閉原則的目標。8 8關於 SOLID 原則，詳細請參考大澤木小鐵的從實例學習設計模式 威力加強版 使用PHP

PhpStorm 會讓你預覽即將重構的變更，按 Do Refactor 繼續。
